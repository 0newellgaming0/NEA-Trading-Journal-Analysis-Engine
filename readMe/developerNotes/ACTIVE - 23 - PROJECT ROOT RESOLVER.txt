Module: path_resolver.py (Project Root + Data/DB Path Management)
📌 Overview

This module is a centralized path resolution utility for the entire trading system.
It ensures that all file access (CSV market data, SQLite databases, logs, and ingestion pipelines) is:

Consistent
Portable across machines
Independent of runtime working directory
Easy to refactor without breaking imports or file access logic

The module defines a single source of truth for filesystem structure, preventing hardcoded paths scattered across the project.

🧠 Core Design Philosophy

This module is built around three principles:

1. Project Root Awareness

Instead of relying on cwd (current working directory), the system always resolves paths relative to:

journal/  ← TRUE PROJECT ROOT

This prevents:

Broken paths when launching from different scripts
IDE vs terminal inconsistencies
Deployment path issues
2. Separation of Data Types

The system distinguishes between:

📊 Market data (CSV files)
🗄️ Structured databases (SQLite)
🔄 Ingestion/logging systems

Each category has its own controlled access path.

3. Centralized Control Layer

All modules (UI, engines, scrapers, journal system) MUST use this resolver instead of constructing paths manually.

⚙️ Functions
🧩 1. get_project_root()
def get_project_root():
    return os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
🔍 Purpose

Returns the absolute root directory of the project.

📌 How it works
__file__ → current module file
os.path.abspath() → converts to full path
dirname() twice → moves up two levels:
path_resolver.py
modules/
→ journal/ (root)
📤 Output Example
E:\work\NEA28v1\journal
🚨 Why it matters

This prevents fragile code like:

open("../../data/file.csv")
