📌 Renderer Module — Institutional Event Journal Formatter
In-Depth Technical README
Overview

The Renderer Module is the final presentation layer in the institutional trading architecture.

It is responsible for:

Converting structured event, context, trade, and regime data into a human-readable institutional journal narrative, while simultaneously providing full debug-level trace logging.

This module is NOT analytical.

It does NOT compute signals.

It does NOT modify data.

It is a:

pure visualization + debugging + audit layer

🧠 Core Philosophy

The renderer is built on three strict principles:

1. Zero Intelligence Policy

The renderer performs no interpretation of market data.

It only formats pre-computed structures:

event → already detected
context → already analyzed
trade → already planned
regime → already classified
2. Dual Output System

The renderer produces two outputs:

🟡 A. Debug Trace (Logging Layer)
Full structured inspection of input data
Used for auditing, backtesting, and system verification
🟢 B. Journal Prompt (UI Layer)
Clean human-readable institutional report
Used by UI, journal system, CSV exports, and reports
3. Transparency Over Abstraction

Every component is explicitly logged:

no hidden transformations
no implicit logic
no silent mutations

📦 Function Overview
Main Function
format_event_journal_prompt(result: dict)
Input Contract

The renderer expects a structured dictionary:

{
    "event": {},
    "trade": {},
    "context": {},
    "regime": "string"
}
Safety Handling

Each component is safely extracted:

event = result.get("event", {}) or {}
trade = result.get("trade", {}) or {}
context = result.get("context", {}) or {}
regime = result.get("regime", "UNKNOWN")

This ensures:

no NoneType crashes
safe missing-field handling
consistent rendering behavior
