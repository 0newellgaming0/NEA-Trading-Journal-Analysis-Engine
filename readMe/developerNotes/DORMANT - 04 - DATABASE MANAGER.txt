journal_db.py — README.txt (In-Depth Module Documentation)
Overview

This module provides a lightweight SQLite-backed persistence layer for a trading journal system. It is responsible for:

Creating and maintaining the journal database file
Initializing the journal_entries schema
Providing a safe database connection utility
Structuring trade journaling data for multi-ladder execution strategies

It is designed as a core infrastructure module used by higher-level trading journal UI systems (Tkinter dashboards, analytics engines, and trade management tools).

Core Purpose

The module defines a persistent trade journal database that stores:

Trade metadata (ticker, timestamp, account size)
Risk management parameters
Multi-entry ladder execution pricing
Position sizing allocations
Narrative-based trade documentation (analysis, notes, management logic)

This supports an institutional-style journaling workflow, enabling structured review of trade decisions over time.

File Location / Storage Design
data/journal.db
Key Design Decision:
The database is stored inside a local data/ directory
The directory is auto-created if it does not exist
This ensures portability and eliminates external setup requirements

Function: get_connection()
def get_connection():
Purpose

Creates and returns a new SQLite connection to the journal database.

Behavior

Ensures the data/ directory exists:

os.makedirs("data", exist_ok=True)

Opens a SQLite connection:

sqlite3.connect(DB_PATH)
Returns
sqlite3.Connection object
Design Notes
Connection is not pooled
Each call creates a fresh connection
Suitable for:
lightweight GUI apps
low-frequency journaling writes
Not optimized for high-throughput concurrent systems

Database Schema: journal_entries
Table Definition
CREATE TABLE IF NOT EXISTS journal_entries
Column Architecture

The schema is designed for multi-layered trade journaling, combining:

Trade execution data
Risk management structure
Ladder entry system
Qualitative trade analysis
1. Identity & Metadata Fields
Column	Type	Description
id	INTEGER PRIMARY KEY AUTOINCREMENT	Unique journal entry ID
timestamp	TEXT	Trade or journal creation time
ticker	TEXT	Asset symbol (e.g., AAPL, TSLA)
2. Account & Risk Layer
Column	Type	Description
account	REAL	Account size at time of trade
risk_dollar	REAL	Absolute dollar risk per trade
stop	REAL	Stop-loss price level
3. Ladder Entry System (Scaled Positioning)

This module supports multi-stage institutional scaling entries.

Ladder 1
Column	Type
ladder_1_price	REAL
ladder_1_shares	REAL
ladder_1_total	REAL
Ladder 2
Column	Type
ladder_2_price	REAL
ladder_2_shares	REAL
ladder_2_total	REAL
Ladder 3 / 4 (Advanced Scaling Phase)
Column	Type
ladder_3_price	REAL
ladder_4_price	REAL

Note: These are price-only fields, implying later-stage discretionary scaling or conditional execution logic handled elsewhere.

4. Market Entry Execution Layer
Column	Type	Description
buy_now_price	REAL	Immediate execution price
buy_now_shares	REAL	Shares purchased immediately
buy_now_total	REAL	Total capital deployed instantly
5. Qualitative Analysis Layer

This is where the system transitions from quantitative trading to narrative-based decision tracking.

Column	Type	Description
trade_notes	TEXT	Raw trade commentary
analysis_notes	TEXT	Pre-trade or technical analysis
management_notes	TEXT	Post-entry management decisions
ARCHITECTURAL INTENT

This schema is intentionally designed to support:

1. Institutional-Style Journaling

Captures both:

Execution data (what happened)
Reasoning data (why it happened)
2. Ladder-Based Position Scaling

Supports:

Partial entries
Staged accumulation
Risk distribution across entries
3. Trade Lifecycle Tracking

The three note fields represent a full trade lifecycle:

Analysis phase → analysis_notes
Execution phase → trade + ladder fields
Management phase → management_notes
